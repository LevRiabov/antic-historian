## Design decisions made in this plan

These are baseline-shaping calls, not gate decisions; each is reversible and noted here
so the case study can cite them.

1. **Retrieval has one implementation, shared by evals and API.** The SQL dense-retrieve
   that produced the locked baseline moves from `evals/` into `retrieval/dense.py`; the
   eval harness and the API both call it. Rationale: rag-historian's biggest tax was
   measurement bugs ‚Äî eval/serving drift is the same disease. A parity test pins it:
   harness and API return identical chunk IDs for fixture queries.
2. **LlamaIndex enters behind the thin interface, retrieval-side adapter not replacement**
   (per [ADR-001](docs/adr/001-d1-framework.md)). Our `dense.py` gets a small
   `BaseRetriever` adapter so LlamaIndex synthesis/orchestration can sit on top, but the
   ranked list the eval measured is byte-for-byte what the API serves. No LlamaIndex
   types cross into `api/`, `evals/`, or pydantic response models; models passed
   explicitly, never global `Settings`.
3. **Citations are derived from retrieval metadata, not generated by the LLM.** The model
   emits inline markers `[n]`; the mapping `n ‚Üí {chunk_id, work, author, locator,
   char_start, char_end}` comes from the retrieval result. The LLM can *misplace* a
   marker (faithfulness judge catches that) but cannot *invent* a source.
4. **Sources-first SSE.** Event order: `sources` (the citation table, sent before
   generation starts) ‚Üí `delta` (answer tokens) ‚Üí `done` (usage, latency, cost).
   Frontend can render the source panel immediately; matches the module-10 UX
   North Star.
5. **Provisional D5 lineup for this phase** (config only, nothing hardcoded):
   answer model = local gemma-12b via llama-swap (OpenAI-compatible, already running);
   judge = a hosted frontier model, config-switchable. Judge runs at the phase boundary
   per the cost policy (retrieval metrics stay the free iteration loop).

## Workstreams

### 3.1 Retrieval module (`ahx/retrieval/dense.py`)

- `RetrievedChunk` pydantic model: `chunk_id, work_id, author, work_title, locator,
  text, score, char_start, char_end`.
- `dense_retrieve(query, top_k)` promoted from `evals/retrieval.py`; **sync and async**
  variants (async = pgvector query via SQLAlchemy `create_async_engine` ‚Äî same
  `postgresql+psycopg` URL, psycopg3 drives both; analogy: one pg client, callback vs
  promise API).
- Eval harness refactored to import from here. Re-run `ahx eval run` and confirm
  numbers are identical to the locked baseline (refactor-safety check, logged).
- New parity test: fixture queries ‚Üí same chunk IDs from sync (harness) and async (API)
  paths.

### 3.2 LLM access layer (`ahx/llm.py`)

- `ChatModel` Protocol (‚âà a TS structural interface ‚Äî implementors don't declare it):
  `stream(messages) -> AsyncIterator[str]`, `complete(messages) -> str`, plus a usage
  accessor (prompt/completion tokens).
- One implementation to start: OpenAI-compatible endpoint (`base_url`, `model`,
  `api_key` from `Settings`) ‚Äî covers llama-swap locally and most hosted providers
  unchanged. D5 stays open; business logic sees only the Protocol.
- Settings additions: `chat_base_url`, `chat_model`, `judge_base_url`, `judge_model`,
  API keys via env.

### 3.3 Prompt assembly + citation contract (`ahx/generation/`)

- Context builder: top-k `RetrievedChunk`s rendered as numbered source blocks
  (`[1] Suetonius, Life of Caesar ¬ß82: ‚Ä¶`), deterministic ordering.
- Baseline prompt: answer **only** from sources, cite inline with `[n]`, refuse
  explicitly when sources don't contain the answer (exercised by the 10 OOS questions).
  Prompt text is versioned (`prompt_version` in run records) ‚Äî prompt changes are
  ablations like everything else.
- Pydantic models where data crosses the boundary: `Citation`, `AskRequest`,
  `SourcesEvent`, `DeltaEvent`, `DoneEvent`.
- Marker extraction: parse `[n]` from final text, validate n exists in the source table;
  dangling markers counted, not crashed on (it's a quality signal the judge consumes).

### 3.4 API endpoint (`ahx/api/`)

- `POST /ask` `{question, top_k?}` ‚Üí SSE stream (`sse-starlette`, already a dep):
  `sources` ‚Üí `delta`* ‚Üí `done`.
- **Async end-to-end** (hard rule 7): async embed ‚Üí async retrieve ‚Üí async LLM stream.
  No sync DB/HTTP anywhere in the route.
- Tests: httpx `ASGITransport` + mocked `ChatModel` and embedding transport ‚Äî full SSE
  event sequence asserted without GPU/DB (‚âà supertest with mocked fetch).

### 3.5 Generation-tier eval (`ahx/evals/generation.py`)

The piece that makes the exit criterion measurable. Two layers:

- **Mechanical (free, runs always):**
  - *citation span hit* ‚Äî does any cited chunk contain a gold-span midpoint (same hit
    rule as retrieval tier);
  - *citation precision* ‚Äî fraction of emitted markers that resolve to a gold-covering
    chunk;
  - *refusal accuracy* ‚Äî refused on OOS? answered on in-scope? (string/regex contract on
    the refusal phrasing, kept honest by judge spot-checks);
  - latency + token counts per question (query-time cost is the expensive kind ‚Äî track
    from day one).
- **Judge (phase boundaries):** faithfulness (claims supported by cited sources) and
  completeness (vs `ideal_answer`), 1‚Äì5 rubrics ported from rag-historian's methodology,
  judge model + rubric version recorded in the run record. Known footgun: weak judges
  miscalibrate ‚Äî use a frontier judge from day one.
- `GenerationRun` pydantic record mirroring `RetrievalRun`; saved to
  `backend/evals/runs/`. New CLI: `ahx eval generate [--judge/--no-judge]`.

### 3.6 Lock the baseline

- Full run over all 72 questions (judge on). Append the entry to
  [docs/eval-log.md](docs/eval-log.md): per-category faithfulness/completeness,
  citation precision, refusal accuracy, p50/p95 latency, tokens per answer.
- Case-study assets captured now: raw SSE trace of one answer, a clean refusal on an
  OOS question, the scorecard table.

## Out of scope (resist the pull)

- Rerank, hybrid/BM25, contextual retrieval, chunking re-study ‚Äî **Phase 4**, through
  the ablation door.
- Agent loop / query decomposition ‚Äî **Phase 5** (multi-hop's 6.7% recall@5 stays bad
  in this phase; that's the point of a baseline).
- Caching, routing, rate limits, Langfuse wiring ‚Äî **Phase 6**.
- Frontend ‚Äî **Phase 7**. `/ask` is consumed via `curl`/docs UI for now.
